### Задание 1. Резервное копирование

### Кейс
Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования. 

Необходимо описать, какие варианты резервного копирования подходят в случаях: 

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.
``` Подходящие методы:
Полное резервное копирование (Full backup) раз в сутки.
Использование pg_dump/pg_dumpall (PostgreSQL) или mysqldump (MySQL).
Хранение копий на отдельном сервере/хранилище.

Почему подходит:
Полный backup содержит все данные целиком, позволяет вернуть состояние на конец
предыдущего дня — это ровно то, что требуется.
```
1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.
``` Подходящие методы:
Инкрементные бэкапы + архивирование WAL (PostgreSQL) или binary logs (MySQL).
Point-In-Time Recovery (PITR).

Схема:
Раз в сутки — полный бэкап.
Каждые N минут — инкрементальный бэкап/лог транзакций.
В случае аварии — отматываем лог до времени аварии.

Результат: Данные можно вернуть примерно до последнего часа (или меньше, если логирование чаще).
```

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.

```
Да, это возможно.
Для этого существуют механизмы:
Failover к реплике (standby/slave).
DRBD или Ceph синхронное хранение.
Кластер PostgreSQL Patroni + HAProxy.
MySQL Group Replication / Percona XtraDB Cluster.

Что происходит:
Основной сервер умер.
Реплика автоматически повышается в мастер.
Приложение практически не замечает простоя (<10 секунд).
```

### Задание 2. PostgreSQL

2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).
Пример полного бэкапа базы:
```bash
pg_dump -U postgres -Fc mydb > /backup/mydb.dump
```
Где:
-Fc — формат Custom (подходит для restore).
mydb — имя базы.

Команда восстановления из бэкапа:

```bash

pg_restore -U postgres -d mydb --clean --create /backup/mydb.dump
```

Параметры:
--clean — удаляет объекты перед восстановлением
--create — создаёт базу заново.

2.1.* Возможно ли автоматизировать этот процесс? Если да, то как?

Да, несколько вариантов:

1. Cron + pg_dump:
``` bash
0 2 * * * pg_dump -U postgres mydb > /backup/daily.dump
```
2. Ansible playbook

``` Автоматизация создания директории, пользователя, задач cron. ```

3. Bash-скрипты + ротация бэкапов
```
Удаление старых бэкапов → логирование → уведомления.
```

4. Современные решения:

```
pgBackRest
Barman
Wal-G
Patroni + WAL логирование
```

### Задание 3. MySQL

3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL. 

Первая полная копия:
``` xtrabackup --backup --target-dir=/data/full ```

Далее инкрементный:
``` xtrabackup --backup --target-dir=/data/inc1 --incremental-basedir=/data/full ```

3.1.* В каких случаях использование реплики будет давать преимущество по сравнению с обычным резервным копированием?
```
Реплика лучше обычного бэкапа, если нужно:

1. Высокая доступность
При падении мастера → slave принимает нагрузку.

2. Снятие нагрузки с мастера
SELECT запросы можно слать на реплику (отчёты, BI-системы, аналитику).

3. Быстрое восстановление после аварии
Не ждём восстановления из бэкапа → переключаемся на slave.

4. Тестирование
Реплика идеально подходит для разработки и тестов без риска.

5. Zero-downtime миграции
Обновление версии MySQL без остановки бизнеса.
````
